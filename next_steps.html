Critical Gaps — Blocks Everything

Critical

Three agents are NotImplementedError stubs

agents/planner_agent.py, agents/coder_agent.py, and agents/reviewer_agent.py all raise NotImplementedError from their run() method. Calling graph.invoke() on any real problem will crash immediately. This is the only blocker to running the system.

Required Gaps — Assignment Will Be Incomplete Without These

Required

No single-agent baseline

The assignment specification explicitly states: "Απαιτείται σύγκριση με single-agent baseline στα ίδια benchmarks." There is no SingleAgent class, no --mode single flag in main.py, and no mechanism to run the benchmark without the full 4-agent loop. Without this, the comparison section of the report has no data.

Required

No actual benchmark results

The report requires analysis tables and charts comparing multi-agent vs single-agent performance (pass rate, iterations, token cost). None of this can be produced until real runs are executed and saved to results/.

Required

README.md is empty

The current README.md contains only the project title. For submission the README must explain how to install dependencies, set up the .env, download the dataset, and run both modes. Reviewers need this to reproduce results.

Optional Gaps — Nice to Have

Optional

orchestration/orchestrator.py is unused

The file implements a manual orchestration loop that duplicates the LangGraph pipeline. It is not called from anywhere. Either integrate it as a non-LangGraph comparison point or remove it before submission to avoid confusion.

Optional

No rate-limit retry wrappers

tenacity is in the Pipfile but not used anywhere. A full benchmark run of 20+ problems will hit OpenAI rate limits. Wrapping each LLM call with @tenacity.retry(wait=wait_exponential(min=4, max=60)) prevents crashes.

Optional

pytest output parsing is fragile

TesterAgent currently counts " PASSED" and " FAILED" substrings in pytest stdout. This breaks if pytest output format changes. Consider adding pytest-json-report for structured, reliable output.

Dependency on .env API Key

You need a working API key before anything can run. Copy .env.example to .env and fill in OPENAI_API_KEY (for GPT-4o) or ANTHROPIC_API_KEY (for Claude). Without this, all three LLM agent stubs will fail at the LLM client constructor, not just at the API call.

    Each follows the same pattern: construct a prompt from task + context, call the LLM, return an AgentResult. Use ChatOpenAI from langchain_openai (already in Pipfile).

    from langchain_openai import ChatOpenAI
    from langchain_core.messages import SystemMessage, HumanMessage

    def run(self, task: str, context: dict | None = None) -> AgentResult:
      llm = ChatOpenAI(model=self.model, temperature=0)
      messages = [SystemMessage(content=self.system_prompt)]
      # build user message from task + context fields
      messages.append(HumanMessage(content=self._build_prompt(task, context)))
      response = llm.invoke(messages)
      return AgentResult(
        agent_name=self.name,
        output=response.content,
        success=True,
        tokens_used=response.usage_metadata["total_tokens"],
      )

What the Baseline Needs to Do

The assignment requires: "Σύγκριση απόδοσης πρακτόρων/επαναλήψεων με ποιότητα αποτελέσματος και κόστος (API calls)." A single-agent baseline is the simplest possible system: one LLM call to write code, one test run, no planning, no review, no loop.

The baseline does NOT need to be a separate codebase. Add a --mode single flag to the existing main.py. The same benchmark loader, TesterAgent, MetricsCollector, and output format can all be reused.

Minimal Implementation Sketch

# agents/single_agent.py — no LangGraph needed

class SingleAgent(BaseAgent):
  def run(self, task, context=None):
    llm = ChatOpenAI(model=self.model, temperature=0)
    prompt = (
      f"Solve the following Python programming problem.\n"
      f"Return ONLY the Python function, no explanation.\n\n{task}"
    )
    response = llm.invoke([
      SystemMessage(content="You are an expert Python developer."),
      HumanMessage(content=prompt),
    ])
    return AgentResult(
      agent_name=self.name,
      output=response.content,
      success=True,
      tokens_used=response.usage_metadata["total_tokens"],
    )

What to Compare in the Report

Metric What it shows
pass@1 How often the first attempt is correct — single agent's strongest metric
Overall pass rate Average fraction of tests passing — multi-agent should win here due to feedback loop
Avg iterations How many loops were needed — always 1 for single, >1 for multi
Total tokens API cost proxy — single agent is far cheaper per problem
Improvement per iteration Does the multi-agent system actually improve on retries? Plot pass rate vs iteration number
Key research question for the report: Is the quality improvement from the multi-agent loop worth the extra token cost? At what problem difficulty level does the single agent fail and the multi-agent succeed? These are the interesting findings — not just the final numbers.
Expected result: Multi-agent should have a higher overall pass rate on harder problems. Single-agent should win on pass@1 efficiency (cost per correct solution on easy problems). The report should discuss this trade-off, not just report one wins over the other.
