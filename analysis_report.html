Multi-Agent vs Single-Agent Code Generation

HumanEval Benchmark — LangGraph Pipeline Evaluation

Model: gpt-3.5-turbo Dataset: HumanEval (100 problems each) Framework: LangGraph Course: Intelligent Agent Systems & LLMs

Research Question

Central question: When and why does collaboration among multiple specialized agents outperform a single general-purpose agent — and at what cost?

This report evaluates a 4-agent LangGraph pipeline (PlannerAgent → CoderAgent → TesterAgent → ReviewerAgent) against a single-agent baseline on the HumanEval dataset. Both runs used 100 randomly sampled problems with gpt-3.5-turbo. Because problems are randomly sampled, per-problem comparisons are illustrative; aggregate metrics are the primary basis for conclusions.

Task Decomposition — When Planning Adds Value

The PlannerAgent helps most on algorithmically complex problems where a structured plan prevents obvious logical errors on the first attempt. For simple problems (e.g., HumanEval/2, /30, /22 — all solved in 1 iteration), the planner adds tokens with no benefit. This suggests an adaptive pipeline that skips planning below a complexity threshold would meaningfully reduce cost.

Regression cases: HumanEval/99, /123, /138 were each solved by single-agent in one shot but failed after 10 multi-agent iterations. The feedback loop made them worse, not better.

TesterAgent as a Deterministic Oracle

The TesterAgent runs pytest in a sandboxed tempfile.mkdtemp() subprocess with no LLM calls. This makes test evaluation ground-truth accurate and reproducible, unlike LLM-judged evaluation. Each problem runs in a fresh temporary directory, preventing import cache contamination across problems.

Tradeoff: deterministic accuracy, but the system cannot handle problems where the test itself is ambiguous or partial credit is meaningful.

ReviewerAgent Keyword Protocol

The ReviewerAgent uses a binary keyword protocol: the string APPROVED in its response triggers graph termination; anything else is treated as a bug list and passed back to CoderAgent.

Limitation: binary signal lacks nuance — the reviewer cannot express partial approval or confidence levels, which can cause premature rejection of nearly-correct solutions.

Timeout Handling

A subprocess.TimeoutExpired handler was added to the TesterAgent after HumanEval/39 produced an infinite loop that crashed the entire benchmark run. Timeouts (30s limit) are now captured as failed test results and returned as feedback to the CoderAgent, allowing the run to continue on subsequent problems.

Feedback Loop Mechanics

On each retry, CoderAgent receives all of the following signals simultaneously:

SignalSourceContent
Problem descriptionOriginal inputUnchanged across iterations
PlanPlannerAgentStep-by-step implementation guide

Test outputTesterAgent (pytest)Exact failure messages and stack traces
ReviewReviewerAgentNumbered bug list or APPROVED

When feedback signals are precise (pytest output), recovery is possible. When they are vague or incorrect (reviewer hallucinations), the loop compounds errors.

Random Problem Sampling

Problems are randomly sampled from HumanEval on each run using random.sample() (without replacement), ensuring no duplicates. If --max-problems equals the full dataset size (164), all problems are returned in random order.

Task Decomposition

The Planner → Coder → Tester → Reviewer chain implements hierarchical task decomposition: a complex problem ("write a correct Python function") is broken into planning, implementation, verification, and quality assurance sub-tasks, each handled by a specialized agent.

The key insight from the results is that decomposition only adds value when the sub-tasks are genuinely distinct in difficulty — otherwise it introduces overhead without benefit.

Feedback Loops

The Tester→Coder and Reviewer→Coder feedback loops implement closed-loop control: the system measures actual output against desired output and corrects iteratively. This is analogous to PID control in classical engineering.

The results confirm the theoretical prediction: the loop works when the error signal is precise and actionable (pytest output), and fails when the corrective signal is noisy or incorrect (reviewer hallucinations). Signal quality determines loop value.

Emergent Behavior

HumanEval/125 (7 iterations) and HumanEval/12 (8 iterations) demonstrate emergent behavior: neither a single forward pass nor a single review would have produced a correct solution, but the iterative agent interaction converged on one. The correct solution emerged from the system as a whole, not from any individual agent.

These cases are the strongest argument for multi-agent collaboration — outcomes that are structurally impossible for a single-pass system.

When Multi-Agent Outperforms

  • 1The problem is complex enough that planning provides genuine decomposition value
  • 2The first code attempt is nearly correct — a small, identifiable bug exists
  • 3The feedback signal (test output) is precise and actionable enough for the coder to act on

When Multi-Agent Does Not Outperform

  • 1The problem is straightforward — single agent solves it in one shot (64% of cases)
  • 2The underlying model lacks the capability to fix the identified bug regardless of iterations
  • 3The planner or reviewer introduces incorrect guidance that misdirects the coder

Summary Findings

Cost vs Accuracy

+1% solve rate at 31.4× token cost. Multi-agent is not cost-effective for gpt-3.5-turbo at this scale.

Loop Value Is Real

~6 problems recovered that single-agent failed. Value is narrow but genuine — limited to near-miss cases.

Expensive Failures

27/100 problems consumed full 10-iteration budget and still failed. Early stopping heuristics are needed.

Regression Risk

3 problems that single-agent solved became multi-agent failures. Planner/reviewer can degrade performance.

First-Attempt Dominance

64% of multi-agent problems solved on iteration 1 — the feedback loop provides no value for the majority.

Emergent Behavior Confirmed

HumanEval/125 and /12 demonstrate outcomes impossible for single-pass systems — 7 and 8 iterations respectively.

Recommendations for Future Work

  • 1
    Adaptive pipeline: skip PlannerAgent for low-complexity problems based on description length or keyword heuristics — reduces tokens without sacrificing quality
  • 2
    Early stopping: add confidence scoring from ReviewerAgent to exit before iteration 10 when recovery is unlikely
  • 3
    Stronger model: repeat experiment with gpt-4o-mini, gpt-4o, or claude-sonnet — the feedback loop likely shows greater benefit with higher model capability
  • 4
    Larger sample: full 164-problem run for statistical significance — current ±5% confidence interval is too wide for strong conclusions
  • 5
    Cost-weighted metric: report "tokens per solved problem" as the primary metric alongside solve rate to capture the accuracy–cost trade-off
