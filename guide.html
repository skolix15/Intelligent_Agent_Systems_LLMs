Project 12 — Multi-Agent Software Engineering: Developer Guide

Architecture, data flow, agent design, and how to run the system · updated 2026-05-09

What Is This Project?

A multi-agent system where specialised LLM agents collaborate to solve Python programming problems from the HumanEval benchmark. Four agents — Planner, Coder, Tester, Reviewer — form a feedback loop that iteratively writes, tests, and improves code until all tests pass or the iteration budget is exhausted.

A single-agent baseline is also included: one LLM call, no loop. The two modes run on the same benchmark so results are directly comparable.

Status (2026-05-09): All four agents are fully implemented with real LLM calls. HumanEval tests are automatically adapted to pytest format. Single-agent baseline is complete. 11/11 unit tests pass without an API key. The system is ready to run — only a valid API key is required.

Key Research Question

When and why does a multi-agent feedback loop outperform a single-shot LLM call?

  • On easy problems the single agent often solves them first try — cheaper.
  • On harder problems the feedback loop catches bugs and iterates to a correct solution.
  • The report should quantify this trade-off: quality improvement vs. token cost.

Shared State (AgentState TypedDict)

All nodes read from and write to a single immutable-style state dict. LangGraph merges the returned partial dict into the current state after each node.

Stopping Conditions (graph/nodes.py:should_continue)

def should_continue(state): total = state["test_passed"] + state["test_failed"] + state["test_errors"] pass_rate = state["test_passed"] / total if total > 0 else 0.0 if pass_rate >= state["pass_threshold"]: return "end" # all tests pass → done if state["iteration"] >= state["max_iterations"]: return "end" # budget exhausted → done return "continue" # loop back to CoderAgent

LLM Provider Selection (utils/llm_client.py)

The get_llm(model) function transparently returns the right LangChain client: ChatAnthropic when the model name starts with claude-, ChatOpenAI for everything else. Both providers are in the Pipfile.

def get_llm(model: str): if model.startswith("claude"): from langchain_anthropic import ChatAnthropic return ChatAnthropic(model=model, temperature=0) from langchain_openai import ChatOpenAI return ChatOpenAI(model=model, temperature=0)

All three LLM agents also wrap their run() with a tenacity retry (wait_exponential min=4 max=60, 3 attempts) to handle API rate limits gracefully.

Agent Roles and Responsibilities

LLM Agent

PlannerAgent — agents/planner_agent.py

Input: problem description (function signature + docstring)
Output: numbered step-by-step implementation plan (no code)
Prompt strategy: system prompt instructs it to focus on algorithm choice, data structures, and edge cases — explicitly told NOT to write code.
State field written: plan

LLM Agent

CoderAgent — agents/coder_agent.py

Input: problem description, plan, (on retries) previous code + test output + review
Output: clean Python code inside a ```python``` fence
Code extraction: _extract_code() strips the markdown fence. Falls back to raw text if no fence is found.
Retry awareness: on iteration 2+ the prompt includes the previous code, failing test output, and reviewer feedback — giving the model all context needed to fix bugs.
State field written: code, iteration

No LLM — Deterministic

TesterAgent — agents/tester_agent.py

Input: generated Python code + pytest-ready test file
Output: passed/failed/error counts + raw pytest output
Mechanism: writes code as solution.py and tests as test_solution.py to a tempfile.TemporaryDirectory(), runs python -m pytest … --tb=short in a subprocess, parses PASSED / FAILED / ERROR counts from stdout+stderr. Timeout: 30 seconds. Zero LLM cost.
State fields written: test_output, test_passed, test_failed, test_errors

LLM Agent

ReviewerAgent — agents/reviewer_agent.py

Input: problem, plan, code, test results (passed/total + raw output)
Output: either the single word APPROVED or a short, actionable bullet list of specific bugs to fix.
Approved detection: "APPROVED" in result.output.upper() — case-insensitive so model variations don't cause false negatives.
Design note: the reviewer is instructed NOT to rewrite code. It is a critic, not a fixer. Fixes happen in the next CoderAgent iteration.
State fields written: review, approved, history entry

Baseline — No Loop

SingleAgent — agents/single_agent.py

Input: problem description only (no plan, no feedback)
Output: Python code (same fence extraction as CoderAgent)
Purpose: provides the comparison baseline required by the assignment. Uses the same model, TesterAgent, and benchmark so all metrics are comparable.
Used by: run_single_agent() in main.py when --mode single is passed.

How Tests Are Adapted for TesterAgent

HumanEval's built-in tests use a check(candidate) convention — not pytest. The benchmark loader (benchmarks/humaneval.py:_adapt_tests_to_pytest()) converts them automatically when loading the JSONL file:

# HumanEval raw "test" field: def check(candidate): assert candidate([1.0, 2.0, 3.9], 0.3) == True assert candidate([1.0, 2.0], 0.5) == False # After _adapt_tests_to_pytest(raw_test, entry_point="has_close_elements"): from solution import has_close_elements # imports the generated solution def check(candidate): assert candidate([1.0, 2.0, 3.9], 0.3) == True assert candidate([1.0, 2.0], 0.5) == False def test_check(): # pytest discovers and runs this check(has_close_elements) # passes the generated function to the checker

The entry_point field from the JSONL (e.g. has_close_elements) is used both in the import statement and in the test_check() call. This means TesterAgent runs the exact official HumanEval checks against the code generated by CoderAgent — problem-specific, not generic tests.

End-to-End Data Flow (Multi-Agent Mode)

  1. main.py loads data/HumanEval.jsonl via HumanEvalBenchmark. Each Problem has: problem_id, description (prompt), entry_point, canonical_solution, and tests (already adapted to pytest format).
  2. main.py initialises AgentState with problem fields, empty agent outputs, iteration=0, and run config, then calls graph.invoke(initial_state).
  3. PlannerAgent node sends the problem description to the LLM. The response is a plain-text numbered plan. State plan is updated; tokens counted.
  4. CoderAgent node builds a structured prompt from: problem description + plan (+ previous code and feedback on retries). The LLM returns code in a ```python``` fence. Code is extracted, stored in state["code"]; iteration counter incremented.
  5. TesterAgent node writes solution.py (the code) and test_solution.py (the adapted HumanEval tests) to a temp directory. Runs pytest via subprocess. Parses PASSED/FAILED/ERROR counts from output. Updates test_passed / test_failed / test_errors / test_output.
  6. ReviewerAgent node sends problem + plan + code + test summary to LLM. Writes review to state. Sets approved=True if "APPROVED" in response. Appends a history snapshot (code, pass_rate, review, tokens) to state["history"].
  7. should_continue() edge: if pass_rate ≥ pass_threshold or iteration ≥ max_iterations → END. Otherwise → back to CoderAgent (step 4).
  8. main.py receives final state, passes it to MetricsCollector.add_from_state(), then moves to the next problem.
  9. After all problems: aggregate metrics are computed and saved as JSON to the path given by --output.

Prompt Content Per Agent

AgentSystem Prompt SummaryUser Prompt Content

What MetricsCollector Captures

MetricFormulaWhat it reveals
pass@1 Problems solved in exactly 1 iteration / total Single-shot accuracy — single agent's strength
overall_pass_rate Mean of (passed / total tests) across all problems How close solutions get — partial credit
avg_iterations Mean iterations per problem How hard the loop worked; always 1 for single-agent
total_tokens Sum across all problems and all agent calls API cost proxy
