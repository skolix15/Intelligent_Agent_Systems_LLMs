Project 12 — Multi-Agent Software Engineering

Complete Guide · LangGraph + Python · Academic Year 2025–2026

Project Goal

Build a multi-agent system where four specialised AI agents collaborate to solve Python programming problems autonomously. One agent plans, one writes code, one tests it with pytest, and one reviews the result. They run in a feedback loop powered by LangGraph until tests pass or the iteration limit is hit. The system is benchmarked against HumanEval and compared to a single-agent baseline.

Two Testing Layers — Important Distinction

LayerWhat is testedToolWhere
Inside the loop The Python code CoderAgent generated pytest (subprocess) TesterAgent.execute_tests()
Your own system Graph logic, agent wiring, benchmark loader pytest (normal) tests/ folder
Both use pytest — but they are completely separate. The first is part of the agent pipeline; the second is your own unit test suite for verifying the system works correctly before running expensive LLM calls.

Data Flow at a Glance

HumanEval problem → AgentState (init) → graph.invoke() → AgentState (final) → MetricsCollector → results/run.json

LangGraph Feedback Loop

planner_node → coder_node → tester_node → reviewer_node → should_continue() ↩ "continue" → back to coder_node

should_continue() returns "end" when pass_rate ≥ threshold or iteration ≥ max_iterations. Otherwise it returns "continue" and LangGraph routes back to coder_node.

AgentState — the Shared Memory

LangGraph passes a single AgentState dict through every node. Each node reads what it needs and returns an updated copy. Nothing is stored outside this dict during a run.

The Four Agents

Planner

PlannerAgent

Receives the raw problem description. Returns a numbered, step-by-step plan: data structures to use, algorithm choice, edge cases to handle. Called once per problem before the loop starts.

Coder

CoderAgent

Receives the plan + optional test failure output + reviewer feedback. Returns a complete Python function. Called on every iteration of the feedback loop.

Tester

TesterAgent

Writes the generated code and HumanEval's check() function to a temp directory. Runs python -m pytest in a subprocess. No LLM call — pure local execution. Returns pass/fail counts.

Reviewer

ReviewerAgent

Reads the code, the plan, and the test result. Provides feedback on correctness, missing edge cases, and style. Returns APPROVED when the solution is acceptable.

Key insight — TesterAgent is different. The other three agents call an LLM. TesterAgent runs a subprocess. This means it is deterministic, fast, and has zero token cost. The feedback it sends to CoderAgent (the raw pytest output) is far more precise than anything an LLM could invent about test results.

BaseAgent Contract

All four agents inherit from BaseAgent and must implement one method:

class BaseAgent(ABC):
  def __init__(self, name, model, system_prompt): ...

  @abstractmethod
  def run(self, task: str, context: dict | None) -> AgentResult:
    """Call the LLM, return a structured AgentResult."""
    ...

AgentResult carries: output (text), success (bool), tokens_used (int), metadata (dict).

LangGraph — Three Key Files

graph/state.py — AgentState TypedDict

Defines the shape of the shared state dict. LangGraph enforces that every node receives and returns this exact structure. Nothing lives outside it during a run.

from typing import TypedDict

class AgentState(TypedDict):
  problem_id: str
  problem_description: str
  plan: str
  code: str
  test_passed: int
  iteration: int
  ... # full list in state.py

graph/nodes.py — Node functions

Each node is a plain Python function: (AgentState) → AgentState. It calls the relevant agent, updates the state fields it owns, and returns the new state.

def coder_node(state: AgentState) -> AgentState:
  result = _coder.run(
    task=state["problem_description"],
    context={
      "plan": state["plan"],
      "previous_code": state["code"],
      "test_feedback": state["test_output"],
    }
  )
  return {**state, "code": result.output, "iteration": state["iteration"] + 1}

graph/pipeline.py — build_graph()

Creates a StateGraph, registers all four nodes, adds edges, and adds the conditional edge that loops back or ends the graph.

workflow = StateGraph(AgentState)

workflow.add_node("planner", planner_node)
workflow.add_node("coder", coder_node)
workflow.add_node("tester", tester_node)
workflow.add_node("reviewer", reviewer_node)

workflow.set_entry_point("planner")
workflow.add_edge("planner", "coder")
workflow.add_edge("coder", "tester")
workflow.add_edge("tester", "reviewer")

workflow.add_conditional_edges(
  "reviewer", should_continue,
  {"continue": "coder", "end": END}
)

graph = workflow.compile()

TesterAgent — How the Subprocess Works

HumanEval provides a check(candidate) function that asserts against the correct answer. TesterAgent wraps it in a pytest test and runs it in isolation:

# /tmp/xyz/solution.py ← CoderAgent's output
def has_close_elements(numbers, threshold):
  return any(abs(a-b) < threshold for i,a in enumerate(numbers)
                          for b in numbers[i+1:])

# /tmp/xyz/test_solution.py ← HumanEval's check() + pytest wrapper
from solution import has_close_elements
def test_check():
  check(has_close_elements)

# TesterAgent runs:
python -m pytest test_solution.py -v --tb=short

Single-Agent Baseline

The exercise requires a comparison with a single agent. This is just a stripped-down version that skips the planning and review steps — CoderAgent generates code once, TesterAgent runs once, no loop. Add a --mode single|multi flag to main.py to switch between them.

Core Libraries

langgraph ≥ 0.2

The orchestration engine. Provides StateGraph, node registration, edge definitions, and conditional routing. Compiles the graph into a runnable object with a simple .invoke(state) interface.

Core
langchain-core

Base message types (HumanMessage, AIMessage, SystemMessage) and the Runnable protocol that LangGraph builds on. Required as a transitive dependency of LangGraph.

Core
langchain-openai

LangChain wrapper for OpenAI's API. Provides ChatOpenAI which each agent uses to send messages and get responses. Handles retries, streaming, and response parsing.

LLM
langchain-anthropic

Same as above but for Anthropic Claude. Drop-in replacement for ChatOpenAI — swap the class name and the model string, nothing else changes.

LLM
openai

Official OpenAI Python SDK. Used directly inside agent run() methods if you prefer the raw SDK over LangChain's wrapper. Also needed for token counting via tiktoken.

LLM
anthropic

Official Anthropic Python SDK. Alternative to langchain-anthropic for direct API calls to Claude. Useful if you want fine-grained control over messages.

LLM
pydantic ≥ 2.0

Data validation for agent inputs and outputs. Used to validate AgentResult and RunConfig. LangGraph also uses Pydantic internally for state validation.

Core
python-dotenv

Loads OPENAI_API_KEY / ANTHROPIC_API_KEY from a local .env file into environment variables. Called once at the top of main.py with load_dotenv().

Utility
loguru

Structured, colour-coded logging with zero configuration. Replaces Python's built-in logging module. Used to log which agent is running, test results, and iteration counts.

Utility
tiktoken

OpenAI's token counter. Used to count how many tokens each LLM call consumes before making the API request — important for the cost analysis section of the report.

Utility
tenacity

Retry logic with exponential back-off. Wraps LLM API calls so that transient rate-limit errors (429) are automatically retried without crashing the benchmark run.

Utility
pytest dev

Two roles: (1) your unit test runner for the tests/ folder, and (2) the tool TesterAgent invokes in a subprocess against generated code. Both roles use the same binary: python -m pytest.

Dev
pytest-mock dev

Provides mocker.patch() fixtures. Used in integration tests to replace real LLM API calls with mocked responses — so you can test the full agent pipeline without spending API credits.

Dev
pytest-asyncio dev

Adds @pytest.mark.asyncio support. Needed if you later implement async LLM calls (e.g. streaming responses from the OpenAI or Anthropic SDK).

Step-by-Step Build Guide

  1. 1
    Set up environment & API keys

    Copy .env.example to .env and fill in your key. Run pipenv install --dev. Verify with python -c "import langgraph".

    cp .env.example .env
    # edit .env and set OPENAI_API_KEY=sk-...
    pipenv install --dev
    pipenv shell
  2. 2
    Run existing unit tests — they should pass without any API key

    The three test files in tests/ verify the subprocess sandbox, the graph stopping logic, and the benchmark loader. These are the first confidence check before touching any LLM code.

    pytest tests/ -v
    # Expected: 8 passed, 0 failed
  3. 3
    Implement run() in PlannerAgent

    Open agents/planner_agent.py. Replace raise NotImplementedError with an LLM call. Use ChatOpenAI from langchain-openai. Return an AgentResult with the plan text and token count.

    from langchain_openai import ChatOpenAI
    from langchain_core.messages import SystemMessage, HumanMessage

    def run(self, task, context=None):
      llm = ChatOpenAI(model=self.model)
      response = llm.invoke([
        SystemMessage(content=self.system_prompt),
        HumanMessage(content=task),
      ])
      return AgentResult(
        agent_name=self.name,
        output=response.content,
        success=True,
        tokens_used=response.usage_metadata["total_tokens"],
      )
  4. 4
    Implement run() in CoderAgent

    Same pattern as PlannerAgent but the prompt includes context["plan"], context["previous_code"], and context["test_feedback"]. The system prompt instructs the LLM to return only the code block.

  5. 5
    Implement run() in ReviewerAgent

    The prompt passes the generated code + test result summary. The system prompt instructs the LLM to return APPROVED if the solution is correct, otherwise list specific issues.

  6. 6
    Download HumanEval dataset

    Place the JSONL file at data/humaneval.jsonl. Each line is one problem with prompt, canonical_solution, and test fields.

    pip install human-eval
    # or clone: git clone https://github.com/openai/human-eval
    # copy HumanEval.jsonl (after gunzip) to data/humaneval.jsonl
  7. 7
    Run a smoke test on 3 problems

    Verify the full pipeline end-to-end before a full benchmark run. Use --max-iterations 2 to keep costs low.

    python main.py \
      --benchmark data/humaneval.jsonl \
      --max-problems 3 \
      --max-iterations 2 \
      --model gpt-4o \
      --output results/smoke.json
  8. 8
    Add single-agent baseline mode

    Add --mode single|multi to main.py. In single mode, skip the planner and reviewer; CoderAgent runs once, TesterAgent runs once, no loop. Same benchmark, same problems.

  9. 9
    Run full benchmark (20–30 problems)

    Run both modes on the same subset and save separate result files. Use a small subset to control API cost — 20 problems × 5 iterations × 4 agents is roughly 400 LLM calls.

    # Multi-agent run
    python main.py --max-problems 20 --model gpt-4o --output results/multi.json

    # Single-agent baseline
    python main.py --max-problems 20 --model gpt-4o --mode single --output results/single.json
  10. 10
    Analyse results and write report

    Load both JSON files and compare: overall pass rate, pass@1, average iterations, total token cost. Plot pass rate vs number of iterations. Discuss whether more agents always means better results, and where the system fails.

Key Metrics to Report

MetricDefinitionHow to compute
pass@1 Fraction solved on the first code attempt (iteration = 1) solved_in_1 / total_problems
pass@k Fraction solved within k iterations Count problems where iterations_used ≤ k
Overall pass rate Average test pass rate across all problems Mean of test_passed / (passed+failed+errors)
Avg iterations How many feedback loops were needed on average Mean of iteration field in final state
Total tokens Sum of all LLM tokens across all problems (API cost proxy) Sum of total_tokens field across all states

Common Pitfalls

The generated code must be importable. HumanEval's check() calls the function by name (check(has_close_elements)), so the test file must import it from solution.py. Make sure CoderAgent returns only the function definition, not a if __name__ == "__main__" block.
pytest output parsing is fragile. The current parser counts " PASSED" and " FAILED" in stdout. Use pytest --tb=short -q for consistent output, or switch to pytest --json-report (install pytest-json-report) for reliable structured output.
Rate limits will hit during a full run. Wrap every LLM call with tenacity.retry (already in Pipfile). Set wait=tenacity.wait_exponential(min=4, max=60).
Start cheap. Use gpt-4o-mini for development runs and switch to gpt-4o only for the final benchmark. gpt-4o-mini costs ~20× less and is good enough to verify the pipeline works.
