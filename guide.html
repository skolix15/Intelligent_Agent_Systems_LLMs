What This System Does

The system takes a Python programming problem and solves it through an iterative, multi-agent feedback loop. A Planner devises the strategy. A Coder writes the code. A Tester runs the official HumanEval checks via pytest. A Reviewer either approves the solution or sends it back with specific bug reports — triggering another iteration. A Reporter narrates the run and generates an HTML report.

A Single-Agent baseline runs the same benchmark with one LLM call and no feedback loop, providing a direct performance and cost comparison.

Research Question

Does a multi-agent feedback loop outperform a single-shot LLM call — and at what cost?

  • On easy problems, the single agent often solves them first try — lower token cost.
  • On harder problems, the feedback loop catches bugs and iterates toward a correct solution.
  • Metrics (solve rate, pass@1, avg iterations, total tokens) quantify the trade-off.

Stopping Logic — graph/nodes.py · should_continue()

def should_continue(state: AgentState) -> str:
    total     = state["test_passed"] + state["test_failed"] + state["test_errors"]
    pass_rate = state["test_passed"] / total if total > 0 else 0.0

    if pass_rate >= state["pass_threshold"]:
        return "end"       # all tests pass → done

    if state["iteration"] >= state["max_iterations"]:
        return "end"       # budget exhausted → done

    return "continue"       # loop back to CoderAgent

Agents

Agent Roles & Responsibilities

Every LLM agent inherits from BaseAgent, which provides conversation history, token tracking, and the abstract run() interface. The TesterAgent is fully deterministic — no LLM.

BaseAgent — agents/base_agent.py

Abstract base class. Defines the contract and shared infrastructure for all agents.

🏗
BaseAgent (ABC)
agents/base_agent.py

Holds name, model, system_prompt, conversation_history, and total_tokens_used. Provides reset() and _add_message(). Subclasses must implement run(task, context) → AgentResult.

AgentResult carries: agent_name, output, success, tokens_used, iterations, metadata.

PlannerAgent — agents/planner_agent.py

💡 LLM Agent

Role: Software architect. Produces a clear, numbered step-by-step implementation strategy — no code, only a plan.

System prompt: focus on algorithm choice, data structures, edge cases. Explicitly told not to write code.

Input: problem_description
Output → state["plan"]
Retry: Tenacity exponential backoff (3 attempts, 4–60 s)

CoderAgent — agents/coder_agent.py

💡 LLM Agent

Role: Python developer. Writes and iteratively fixes code based on plan, test results, and reviewer notes.

Iteration 1: problem + plan.
Iteration 2+: additionally receives previous_code, test_feedback, review.

Code extraction: strips ```python ... ``` fence. Falls back to raw text if no fence found.

Output → state["code"], state["iteration"]

TesterAgent — agents/tester_agent.py

⚙ Deterministic — No LLM

Role: QA executor. Runs the official HumanEval tests — zero LLM cost, fully reproducible.

  1. Writes solution.py to a fresh temp directory
  2. Writes test_solution.py (adapted HumanEval checks) to same dir
  3. Runs python -m pytest --tb=short via subprocess (30 s timeout)
  4. Parses PASSED / FAILED / ERROR counts from stdout

Output → test_passed, test_failed, test_errors, test_output

How HumanEval Tests Are Adapted

HumanEval uses check(candidate) — not pytest-discoverable. The benchmark loader converts them automatically:

# Original HumanEval test field:
def check(candidate):
    assert candidate([1.0, 2.0, 3.9], 0.3) == True

# After _adapt_tests_to_pytest(raw_test, entry_point="has_close_elements"):
from solution import has_close_elements

def check(candidate):
    assert candidate([1.0, 2.0, 3.9], 0.3) == True

def test_check():               # pytest discovers this
    check(has_close_elements)    # passes the generated function to checker

ReviewerAgent — agents/reviewer_agent.py

💡 LLM Agent

Role: Senior code reviewer. Reads the problem, plan, code, and test results. Decides the next action.

APPROVED — terminates the loop.
Bug list — numbered, actionable issues. Does not rewrite code itself.

Approval detection: "APPROVED" in result.output.upper() — case-insensitive.
Output → review, approved, history entry

ReporterAgent — agents/reporter_agent.py

💡 LLM Agent

Role: Technical writer. After the graph exits, generates an LLM narrative per problem and assembles a self-contained HTML report.

Report includes: aggregate metrics dashboard, per-problem cards with plan, iteration history, final code, and test output. Written to results/report_{mode}_{timestamp}.html.

SingleAgent — agents/single_agent.py

🎯 Baseline

Role: Baseline comparator. One LLM call with only the problem description — no plan, no review, no retry. Same code extraction as CoderAgent, same TesterAgent. Activated by python run.py single.

get_llm(model) — returns ChatAnthropic when name starts with "claude-", otherwise ChatOpenAI. Temperature always 0.

get_tokens_used(response) — reads usage_metadata from LangChain response.

@llm_retry — Tenacity: 3 attempts, exponential wait 4–60 s, skips retry on 401/403.

Data Flow

End-to-End Execution — Multi-Agent Mode

The full pipeline: Planner → Coder → Tester → Reviewer, with a feedback loop between Reviewer and Coder.

  1. Load benchmark

    main.py loads HumanEval.jsonl via HumanEvalBenchmark. Each Problem has: problem_id, description, entry_point, canonical_solution, tests (already pytest-adapted).

  2. Initialise AgentState

    Builds the initial state with problem fields, empty outputs, iteration=0, and run config. Calls graph.invoke(initial_state).

  3. Planner node

    Sends problem description to LLM. Response is a plain-text numbered plan. Updates state["plan"] and state["total_tokens"].

  4. Coder node

    Builds prompt from problem + plan (+ previous code, test output, review on retries). Extracts code from ```python``` fence. Updates state["code"] and increments state["iteration"].

  5. Tester node

    Writes solution.py and test_solution.py to a fresh temp dir. Runs pytest. Parses PASSED / FAILED / ERROR counts. Updates test_passed, test_failed, test_errors, test_output.

  6. Reviewer node

    Sends problem + plan + code + test summary to LLM. Sets approved=True if APPROVED in response. Appends iteration snapshot to state["history"].

  7. Conditional edge — should_continue()

    If pass_rate ≥ pass_threshold or iteration ≥ max_iterations → END. Otherwise → back to step 4 (Coder).

  8. Collect + next problem

    Final state passed to MetricsCollector.add_from_state(). Loop repeats for all problems.

  9. Reporter + save results

    Metrics saved to JSON. ReporterAgent generates results/report_{mode}_{timestamp}.html.

End-to-End Execution — Single-Agent Mode

The baseline: one LLM call, one test run, no loop. Activated by python run.py single.

  1. Load benchmark

    Same as multi-agent: HumanEvalBenchmark loads HumanEval.jsonl. Each Problem carries problem_id, description, entry_point, canonical_solution, and tests (already pytest-adapted).

  2. SingleAgent generates code

    SingleAgent.run(task=problem.description) makes a single LLM call with only the problem description — no plan, no prior feedback. Returns an AgentResult with the extracted Python code and token count.

  3. TesterAgent executes tests

    TesterAgent.execute_tests(code, tests) writes solution.py and test_solution.py to a temp dir and runs pytest. Returns passed, failed, errors, and output.

  4. Build AgentState snapshot

    main.py manually constructs an AgentState dict from the results — iteration=1, max_iterations=1, no plan, no review. This keeps the output format identical to multi-agent for fair metric comparison.

  5. Collect metrics + next problem

    State passed to MetricsCollector.add_from_state(). No retry — loop moves immediately to the next problem.

  6. Reporter + save results

    Metrics saved to results/single.json. ReporterAgent generates results/report_single_{timestamp}.html.

⇄
Why the same AgentState format? Both modes write to the same AgentState TypedDict and use the same MetricsCollector. This means pass@1, solve_rate, and total_tokens are computed identically — making the comparison between single.json and multi.json fair and direct.

Theoretical Background

Core Concepts & How They Are Implemented

The assignment targets three specific concepts from multi-agent systems theory. Each one is defined below alongside how it is realised in both the multi-agent pipeline and the single-agent baseline — the contrast between the two is what makes the concepts measurable.

1

Task Decomposition

Αποσύνθεση Εργασίας

What It Is

Instead of one general agent handling everything, a complex task is broken into smaller, well-defined sub-tasks. Each sub-task is assigned to an agent that is specifically designed — and constrained — to do only that one thing well.

The key benefit: specialisation reduces the cognitive load on each agent and prevents one agent's mistakes from cascading into adjacent responsibilities.

Multi-Agent — Full Decomposition

The problem "write correct Python code" is split into four locked roles:

  • Planner Algorithm strategy only — no code
  • Coder Implementation only — single code block
  • Tester Execution only — subprocess pytest, zero LLM
  • Reviewer Critique only — must not rewrite

Each agent has a distinct system prompt enforcing its boundary.

Single-Agent — No Decomposition

One agent (SingleAgent) is asked to do everything at once: understand the problem, plan an approach, and produce correct code — in a single LLM call. There are no role boundaries. This is the baseline that decomposition is compared against.

2

Feedback Loops

Βρόχοι Ανατροφοδότησης

What It Is

The output of one stage becomes the input of the next, and the system repeats until a measurable quality criterion is met. This is a closed-loop control mechanism — the system can observe its own performance and adjust accordingly.

Without a feedback loop, errors propagate with no correction path. With one, the system can iteratively converge toward a correct solution.

Multi-Agent — Closed-Loop Control

The LangGraph conditional edge implements the loop:

Coder → Tester → Reviewer
                    ↓ should_continue()
               "continue" → Coder (retry)
               "end"      → END

On each retry the Coder receives test_feedback review previous_code. Stops when pass_rate ≥ pass_threshold or iteration ≥ max_iterations.

Single-Agent — Open Loop

There is no loop. The code is generated once, tests are run once, and execution stops regardless of whether the tests pass. Errors are observed but never acted on — this is an open-loop system. The delta in solve rate between the two modes directly measures how much the feedback loop contributes.

3

Emergent Behavior

Αναδυόμενη Συμπεριφορά

What It Is

Behavior that was not explicitly programmed into any individual agent, but arises from their interaction. The system as a whole can do something that none of its parts could do alone.

This is the most theoretically interesting concept — you cannot point to a single line of code that produces it. It appears only when agents interact through the shared state over multiple iterations.

Multi-Agent — Emergent Iterative Debugging

No single agent can debug code autonomously:

  • Planner doesn't know if code will work
  • Coder doesn't run tests
  • Tester doesn't understand the code
  • Reviewer doesn't write code

Together they produce automated iterative debugging — a capability that none holds alone. It is measured as the solve-rate gap between multi.json and single.json.

Single-Agent — No Emergence

With only one agent and no interaction, there is nothing for emergent behavior to emerge from. The single-agent result represents the upper bound of what a single LLM call can achieve without collaboration — and serves as the control group that makes emergence measurable.
